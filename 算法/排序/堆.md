# 堆

## 应用场景
堆（Heap）是一种高效维护动态极值的数据结构，核心优势在于：
1 插入和取出最值的时间复杂度均为 O(log n)，
2 查看最值为 O(1)。
因此，堆广泛应用于需要频繁访问或更新最大/最小元素的场景：
- TOP-K

## 性质：
1 完全二叉树 （最下层的叶都集中再左边，不能有空节点）

2 **引索关系**：
假设：
完全二叉树（Complete Binary Tree）为深度为k、有n个结点的二叉树
(另外，满二叉树的结点数为：n = 2^k -1)
- k = (log2 n)向下取整 + 1;

- 设根结点的引索为0：
- 父结点：(i-1)//2
- 左孩子：2i + 1
- 右孩子：2i + 2


## 常用操作
```python
import heapq
heap = []
heapq.heappush(heap, x)        # 插入
x = heapq.heappop(heap)        # 弹出最小值
x = heapq.heappushpop(heap, item) # 将 item 推入堆中，然后弹出并返回堆中的最小元素。这比单独调用 heappush() 和 heappop() 更高效，具体原理：先与根节点比较，（若小于最小值直接返回item），后插入到尾部，根尾交换，堆化，弹出，实际只进行了一次堆化操作。
x = heap[0]                    # 查看最小值（不弹出
heapq.heapify(arr)             # 将 list 转为堆（O(n)）

heapq.nlargest(n, iterable, key=None)
返回可迭代对象中最大的 n 个元素。key可以根据对象中的某一参数作为对比值

heapq.nsmallest(n, iterable, key=None)
返回可迭代对象中最小的 n 个元素。
```

### 实现原理讲解
建堆 ： 自下向上 从最后一个叶子节点的父节点开始，进行下沉操作 O(n);
(自上而下，维护一个堆列表，每次把新元素插入到最后的位置，对他进行上浮操作，O(nlogn))

插入：插入到最后面，与父节点比较和交换，直至父节点为空或满足堆的条件

弹出：交换根节点和末尾结点，弹出尾节点，堆根结点进行下沉操作

heapq.nlargest 返回可迭代对象中最大的 n 个元素：
设有k(k>n)大小的列表，先取数组中的前n个元素并堆化（最小堆），遍历其他元素，若大于根节点（即大于该n个数的最小值），弹出堆顶最小值，插入到堆顶，堆化。


## 自定义规则的堆排序使用
__lt__：是 heapq 工作的核心引擎。没有它，heapq 不知道如何比较你的对象。
__repr__：是调试助手。它不参与排序逻辑，但让你能看懂堆里到底存了什么。

```pyhton
import heapq

class Task:
    def __init__(self, priority, time, name):
        self.priority = priority
        self.time = time
        self.name = name

    # 核心：自定义 "小于" 的逻辑
    # heapq 问：self < other 吗？
    # 我们定义：如果我的优先级更高，或者优先级相同但时间更早，我就“小于”对方（从而排前面）
    def __lt__(self, other):
        if self.priority != other.priority:
            return self.priority > other.priority  # 优先级大的视为“小”
        return self.time < other.time              # 时间小的视为“小”

    def __repr__(self):
        return f"P{self.priority}-T{self.time}-{self.name}"

heap = []
tasks = [
    Task(5, 10, "A"),
    Task(5, 2, "B"),
    Task(8, 5, "C")
]

for t in tasks:
    heapq.heappush(heap, t)

while heap:
    print(heapq.heappop(heap))
    
# 输出顺序与方法一完全一致，但逻辑封装在类内部，更清晰
```

这两个是 Python 的**魔术方法（Magic Methods）**，也叫双下划线方法。它们让自定义的类能够像内置类型（如数字、字符串）一样工作。

在 `heapq` 的语境下，它们的联动关系非常紧密：
1.  **`__lt__`**：是 `heapq` 工作的**核心引擎**。没有它，`heapq` 不知道如何比较你的对象。
2.  **`__repr__`**：是**调试助手**。它不参与排序逻辑，但让你能看懂堆里到底存了什么。

### 1. `__lt__` (Less Than) —— 排序的灵魂

#### **是什么？**
当你写代码 `a < b` 时，Python 实际上在后台调用 `a.__lt__(b)`。
*   如果返回 `True`，表示 `a` 小于 `b`。
*   如果返回 `False`，表示 `a` 不小于 `b`。

#### **与 `heapq` 的联动**
`heapq` 是一个**最小堆**。为了维持堆的性质（父节点 < 子节点），它在插入 (`heappush`) 和弹出 (`heappop`) 时，会频繁地执行类似这样的比较：
```python
if child < parent:
    # 交换位置，把小的浮上来
    swap(child, parent)
```
如果你放入堆的是自定义对象（比如 `Task` 类），Python 默认不知道如何比较两个 `Task` 谁大谁小。
*   **如果没有 `__lt__`**：程序会报错 `TypeError: '<' not supported between instances of 'Task' and 'Task'`。
*   **如果有 `__lt__`**：`heapq` 会调用你写的逻辑来决定谁该排在堆顶。

#### **代码示例**
```python
import heapq

class Task:
    def __init__(self, priority, name):
        self.priority = priority
        self.name = name

    # 【关键联动点】定义“小于”的逻辑
    # heapq 想要最小的元素在堆顶。
    # 题目要求：优先级数字越大，越优先（越应该排在堆顶）。
    # 所以我们要欺骗 heapq：如果我的 priority 更大，我就假装自己“更小”。
    def __lt__(self, other):
        # 这里的逻辑是：自认为“小”的条件
        return self.priority > other.priority 

t1 = Task(5, "低优先级任务")
t2 = Task(10, "高优先级任务")

heap = []
heapq.heappush(heap, t1)
heapq.heappush(heap, t2)

# 弹出时，heapq 内部比较 t1 和 t2
# 调用 t2.__lt__(t1) -> 10 > 5 返回 True
# heapq 认为 t2 比 t1 "小"，所以 t2 被放在堆顶
best = heapq.heappop(heap)
print(best.name) 
# 输出: "高优先级任务" (正确！)
```

> **注意**：在 Python 3 中，通常只需要实现 `__lt__` 就足够让 `heapq` 工作了。如果需要支持 `<=`, `>`, `>=` 等其他比较，可以使用 `@functools.total_ordering` 装饰器自动补全。

### 2. `__repr__` (Representation) —— 调试的眼睛

#### **是什么？**
`__repr__` 定义了对象的**官方字符串表示形式**。
*   当你在交互式命令行输入变量名回车时。
*   当你打印一个包含该对象的列表、字典或堆时。
*   当你使用 `repr(obj)` 函数时。
都会调用这个方法。

#### **与 `heapq` 的联动**
`__repr__` **完全不参与** `heapq` 的排序逻辑。`heapq` 根本不在乎它的存在。
**但是**，由于堆是一个列表结构，当你想要检查堆的状态（比如 `print(heap)`）时，如果没有 `__repr__`，你会看到一堆晦涩的内存地址，完全无法调试。

#### **对比演示**

**情况 A：没有 `__repr__`**
```python
class Task:
    def __init__(self, p, n):
        self.p = p
        self.n = n
    def __lt__(self, other):
        return self.p > other.p

heap = [Task(10, "A"), Task(5, "B")]
print(heap)
# 输出: [<__main__.Task object at 0x7f8b1c2d3e50>, <__main__.Task object at 0x7f8b1c2d3f10>]
# ❌ 崩溃：完全看不出哪个是哪个，无法调试排序是否正确。
```

**情况 B：有 `__repr__`**
```python
class Task:
    def __init__(self, p, n):
        self.p = p
        self.n = n
    def __lt__(self, other):
        return self.p > other.p
    
    # 【关键联动点】定义打印格式
    def __repr__(self):
        return f"Task(p={self.p}, name='{self.n}')"

heap = [Task(10, "A"), Task(5, "B")]
# 模拟 heapify 过程（虽然这里没调，但假设堆化后）
import heapq
heapq.heapify(heap)

print(heap)
# 输出: [Task(p=10, name='A'), Task(p=5, name='B')]
# ✅ 清晰：一眼看出优先级 10 的任务排在了前面（索引 0）。
```

### 3. 完整实战模板

如果你在做一个复杂的调度系统，这是标准的写法模板：

```python
import heapq
from functools import total_ordering

@total_ordering  # 加上这个，你只需写 __lt__ 和 __eq__，其他比较符自动生成
class Job:
    def __init__(self, job_id, priority, timestamp):
        self.job_id = job_id
        self.priority = priority
        self.timestamp = timestamp

    # 1. 定义排序逻辑 (联动 heapq)
    def __lt__(self, other):
        # 规则：优先级高的先出；若相同，时间戳早的先出
        if self.priority != other.priority:
            return self.priority > other.priority  # 优先级大 -> 视为“小”
        return self.timestamp < other.timestamp    # 时间早 -> 视为“小”

    def __eq__(self, other):
        # 用于判断相等（total_ordering 需要）
        return (self.priority == other.priority and 
                self.timestamp == other.timestamp)

    # 2. 定义打印逻辑 (联动 print/debug)
    def __repr__(self):
        return f"<Job #{self.job_id} | P:{self.priority} | T:{self.timestamp}>"

# --- 测试 ---
jobs = [
    Job(101, 5, 100),
    Job(102, 8, 105), # 优先级最高
    Job(103, 5, 90),  # 同优先级，时间最早
    Job(104, 5, 95)
]

heap = []
for j in jobs:
    heapq.heappush(heap, j)

print("堆的状态:", heap) 
# 输出: [<Job #102 | P:8 | T:105>, <Job #103 | P:5 | T:90>, ...]
# 可以看到 #102 在第一个，#103 在第二个，逻辑正确且可读性强。

print("\n执行顺序:")
while heap:
    job = heapq.heappop(heap)
    print(f"正在处理: {job}") 
    # __repr__ 在这里被隐式调用，输出友好的字符串
```

### 总结

| 方法 | 作用 | 是否被 `heapq` 直接调用？ | 重要性 |
| :--- | :--- | :--- | :--- |
| **`__lt__`** | 定义 **"小于"** 的规则 | **是** (核心依赖) | ⭐⭐⭐⭐⭐ (没有它代码会崩) |
| **`__repr__`** | 定义 **打印/调试** 的格式 | 否 (仅被人眼或 `print` 调用) | ⭐⭐⭐ (没有它能跑，但没法调试) |

**一句话口诀**：
想让堆**跑得对**，必须写 `__lt__`；
想让堆**看得懂**，最好写 `__repr__`。
